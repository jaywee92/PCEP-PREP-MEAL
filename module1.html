📚 Theory: Programming Fundamentals

What is a Compiler?

A compiler is a program that translates the entire source code to machine code at once before the program is executed. Examples: C, C++, Java (to bytecode).

What is an Interpreter?

An interpreter executes the source code line by line and translates it during execution. Examples: Python, JavaScript, Ruby.

Python: A Hybrid System

Python combines both approaches:

  • Compilation: The code is compiled to bytecode (.pyc files)
  • Interpretation: The Python Virtual Machine (PVM) interprets the bytecode

Variables and Data Types

Variable: A named container for a value. In Python, variables are dynamically typed.

Basic Data Types:

  • int - Integers (e.g., 42, -17)
  • float - Decimals (e.g., 3.14, -2.5)
  • str - Strings/Text (e.g., "Hello", 'World')
  • bool - Boolean values (True, False)

Syntax Rules That Are Frequently Tested

  • Indentation is mandatory for code blocks.
  • Code blocks begin with a colon (:).
  • Python is case-sensitive: name and Name are different.
  • Valid identifiers start with a letter or underscore, then letters, digits, underscores.

()  **  +x/-x  * / // %  + -  comparisons  not  and  or

print(2 ** 3 ** 2)  # 512 (right-to-left)

⚠️ Common Mistakes in Module 1

Mistake 1: Confusing Python 2 and 3

Wrong (Python 2):

print "Hello"  # SyntaxError in Python 3!

Correct (Python 3):

print("Hello")  # Correct!

Mistake 2: Confusing Integer Division

In Python 3:

5 / 2 = 2.5   # True Division
5 // 2 = 2    # Floor Division

Remember: In Python 2, 5/2 = 2, but in Python 3, 5/2 = 2.5!

Mistake 3: Invalid Variable Names

Invalid:

123var = 5      # Cannot start with digit
var-name = 10   # Hyphen not allowed
var name = 15   # Spaces not allowed

Valid:

var123 = 5          # OK
_var_name = 10      # OK
varName = 15        # OK (camelCase)

Mistake 4: Type Confusion

Problem:

age = "25"  # String, not integer!
print(age + 5)  # TypeError!

Solution:

age = int("25")  # Convert to int
print(age + 5)   # OK: 30

Mistake 5: Misunderstanding Compiler vs Interpreter

Many think Python is purely interpreted. Actually:

  • Python compiles to bytecode (.pyc)
  • The Python VM (PVM) then interprets this bytecode
  • This is a hybrid approach

Mistake 6: Using is Instead of ==

Wrong:

x = 256
y = 256
print(x is y)   # identity check, not value check

Correct:

print(x == y)   # value comparison
print(x is None)  # 'is' is ideal for None

Mistake 7: Assuming Standard Rounding Rules

Trap: Python uses banker's rounding for .5 cases.

print(round(2.5))  # 2
print(round(3.5))  # 4

Mistake 8: Confusing Keywords with Similar-Looking Names

From raw exam questions: True is illegal as identifier, but TRUE and true_value are legal.

# illegal
for = 10
True = 1

# legal
TRUE = 1
true_value = 10

Mistake 9: Assuming String Input Auto-Formats Itself

Trap from imported sets:

name = input()
print(name == " Robert ")   # False for input: robert

Safer pattern:

name = input().strip().lower()
print(name == "robert")

🚨 Critical Exam Pitfalls

These are the most common traps that catch even experienced programmers in the PCEP exam. Master these to avoid losing easy points!

🚨 Division Always Returns Float
❌ Wrong: 10/2 = 5
✓ Correct: 10/2 = 5.0
The / operator ALWAYS returns a float, even when dividing evenly. Use // for integer division.
🚨 Floor Division with Negatives
❌ Wrong: -7//2 = -3
✓ Correct: -7//2 = -4
Floor division rounds DOWN toward negative infinity, not toward zero. This is often counterintuitive with negative numbers.
🚨 int() Truncates, Doesn't Round
❌ Wrong: int(3.9) = 4
✓ Correct: int(3.9) = 3
The int() function truncates (cuts off) the decimal part, it doesn't round. Use round() for rounding.
🚨 Exponentiation Before Negation
❌ Wrong: -2**2 = 4
✓ Correct: -2**2 = -4
Exponentiation ** has higher precedence than unary minus. -2**2 means -(2**2). Use (-2)**2 for the square of -2.
🚨 Right-to-Left Exponentiation
❌ Wrong: 2**3**2 = 64
✓ Correct: 2**3**2 = 512
Exponentiation is right-associative: 2**3**2 = 2**(3**2) = 2**9 = 512, not (2**3)**2 = 64.
🚨 bool() Truthiness Rules
❌ Wrong: bool('0') = False, bool([0]) = False
✓ Correct: bool('0') = True, bool([0]) = True
Non-empty strings and non-empty lists are always True, regardless of their content. Only empty containers and 0 are falsy.
🚨 Operator Precedence
❌ Wrong: 5 + 3 * 2 = 16
✓ Correct: 5 + 3 * 2 = 11
Multiplication and division have higher precedence than addition and subtraction. 3 * 2 is evaluated first, then 5 + 6 = 11.
🚨 input() Returns String
❌ Wrong: age = input(); if age >= 18: ...
✓ Correct: age = int(input()); if age >= 18: ...
The input() function always returns a string, even for numeric input. You must convert it with int() or float().
🚨 print() Returns None
❌ Wrong: result = print('hello')
✓ Correct: print('hello'); result = 'hello'
The print() function displays output but returns None. It's for side effects, not for returning values.
🚨 String + Number TypeError
❌ Wrong: x = '3' + 3
✓ Correct: x = str(3) + str(3) OR x = 3 + 3
You cannot concatenate strings and numbers directly. Convert both to the same type first with str() or int().
🚨 round() Uses Banker's Rounding
❌ Wrong: round(2.5) = 3
✓ Correct: round(2.5) = 2, round(3.5) = 4
Python uses bankers rounding for midpoint values. This appears in exam code-tracing questions.
🚨 Float Precision Is Not Exact
❌ Wrong: 0.1 + 0.2 == 0.3 is always True
✓ Correct: 0.1 + 0.2 == 0.3 # False
Floating-point numbers are binary approximations. For exact decimal checks, avoid direct equality comparisons.
🚨 Keyword vs Identifier Trap
❌ Wrong: True, for, and are valid variable names
✓ Correct: true_value and TRUE are valid; True/for/and are illegal
Raw PCEP questions repeatedly test reserved words and boolean literals. Similar spelling does not mean same identifier legality.
🚨 String Equality Is Exact
❌ Wrong: input_name == " Robert " will match robert
✓ Correct: Case and spaces must match exactly unless you normalize first
This appears in new OCR-derived items. Normalize user input with strip() and lower() before comparison when needed.
